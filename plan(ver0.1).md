# pdf-pii-masker PDF 개인(신용)정보 마스킹 웹앱 실행계획

## 작업: PDF 업로드 이후 페이지별 이미지 변환, 수동 마스킹, 이미지 기반 PDF 다운로드까지의 MVP 구현 계획

---

## 1. 목표

pdf-pii-masker의 1차 목표는 사용자가 업로드한 PDF를 브라우저에서 페이지별 이미지로 렌더링하고, 각 페이지 위에 수동 마스킹 박스를 추가한 뒤, 마스킹이 반영된 이미지 기반 PDF 사본을 다운로드할 수 있도록 하는 것이다.

최종적으로는 주민등록번호, 사업자등록번호, 이름 등 개인(신용)정보를 자동 탐지하고 마스킹하는 기능을 지향하되, 현재 단계에서는 자동 탐지보다 PDF 처리 파이프라인과 수동 검수·마스킹 기능을 우선 구현한다.

핵심 목표는 다음과 같다.

1. PDF 업로드 후 전체 페이지를 화면에 미리보기로 표시한다.
2. 각 PDF 페이지를 canvas 또는 이미지 기반으로 렌더링한다.
3. 사용자가 페이지 위에 마스킹 박스를 수동으로 추가할 수 있도록 한다.
4. 마스킹 박스를 적용한 상태로 이미지 기반 PDF를 재생성한다.
5. 결과 PDF를 브라우저에서 다운로드할 수 있도록 한다.
6. 원본 PDF와 결과 PDF는 서버에 저장하지 않는다.

---

## 2. 범위

이번 MVP 범위에 포함되는 기능은 다음과 같다.

| 구분 | 포함 내용 |
|---|---|
| PDF 업로드 | 기존 업로드 페이지 유지 |
| PDF 렌더링 | PDF.js를 사용하여 페이지별 canvas 렌더링 |
| 페이지 미리보기 | 업로드한 PDF의 모든 페이지를 순서대로 화면 표시 |
| 페이지 상태 관리 | 페이지 번호, 크기, canvas 참조 또는 이미지 데이터 관리 |
| 수동 마스킹 | 사용자가 드래그로 마스킹 박스 생성 |
| 마스킹 박스 관리 | 페이지별 좌표 저장, 삭제 기능 |
| PDF 재생성 | 마스킹 박스를 canvas에 반영 후 이미지 기반 PDF 생성 |
| 다운로드 | Blob URL 방식으로 브라우저 다운로드 링크 제공 |
| 저장 정책 | 원본·결과 파일 서버 저장 없음 |

---

## 3. 제외범위

이번 단계에서는 다음 기능을 구현하지 않는다.

| 제외 항목 | 제외 사유 |
|---|---|
| OCR | 스캔본 자동 탐지는 후속 단계에서 검토 |
| LLM 판단 | 성명·주소 등 비정형 판단은 후속 단계 |
| 외부 OCR API | 개인정보 외부 전송 및 API 키 관리 이슈로 보류 |
| 외부 LLM API | 개인정보 전송, 비용, 보안정책 검토 전까지 보류 |
| 자동 개인정보 탐지 | 정규식 기반 탐지는 수동 마스킹 MVP 이후 구현 |
| 서버 저장 | 현재 요건상 원본·결과 PDF를 저장하지 않음 |
| 작업 이력 관리 | 보안·감사 정책 확정 전까지 보류 |
| 사용자 계정 기능 | 현재 MVP 범위 아님 |
| 다운로드 파일 영구 보관 | 브라우저 창 종료 후 복구하지 않는 요건 유지 |

### Version 1. 기존 PDF 업로드 페이지 상태 확인

#### 작업 내용

현재 구현된 PDF 업로드 페이지의 구조를 확인한다.

확인 대상은 다음과 같다.

1. `src/App.tsx`의 현재 업로드 UI 구조
2. PDF 파일 선택 이벤트 처리 방식
3. 파일 상태값 관리 방식
4. Vite 개발 서버에서 정상 표시 여부
5. 향후 PDF.js, pdf-lib 적용 가능 구조 여부

#### 완료 기준

- PDF 파일을 선택할 수 있다.
- 선택된 PDF 파일명이 화면에 표시된다.
- 업로드된 파일을 `ArrayBuffer`로 읽을 수 있는 구조가 준비되어 있다.

---

### Version 2. PDF.js 도입 및 페이지별 렌더링 구현

#### 작업 내용

업로드된 PDF를 PDF.js로 로드하고, 전체 페이지를 canvas로 렌더링한다.

처리 흐름은 다음과 같다.

```text
PDF 파일 선택
→ file.arrayBuffer()
→ PDF.js getDocument()
→ 전체 페이지 수 확인
→ 각 페이지 getPage()
→ viewport 생성
→ canvas 렌더링
→ 화면에 페이지별 미리보기 표시
구현 원칙
서버 업로드 없이 브라우저에서 처리한다.
각 페이지는 wrapper div 안에 배치한다.
wrapper div는 향후 마스킹 박스를 overlay할 수 있는 구조로 만든다.
페이지 번호를 화면에 표시한다.
렌더링 scale은 우선 2.0 기준으로 시작한다.
완료 기준
PDF 업로드 후 전체 페이지 수가 표시된다.
모든 페이지가 화면에 순서대로 표시된다.
각 페이지가 이미지처럼 확인 가능하다.
페이지별 wrapper 구조가 생성되어 있다.
Version 3. 페이지별 상태 데이터 구조 설계
작업 내용

각 페이지에 대한 렌더링 정보와 마스킹 박스 정보를 저장할 상태 구조를 설계한다.

예상 구조는 다음과 같다.

type PageRenderState = {
  pageNumber: number;
  width: number;
  height: number;
  scale: number;
  canvasDataUrl?: string;
  masks: MaskBox[];
};

type MaskBox = {
  id: string;
  pageNumber: number;
  x: number;
  y: number;
  width: number;
  height: number;
};
설계 원칙
마스킹 좌표는 페이지별 canvas 기준 좌표로 저장한다.
화면 표시 좌표와 실제 canvas 좌표의 변환 관계를 명확히 한다.
다운로드 시 동일 좌표를 canvas에 반영할 수 있어야 한다.
후속 자동 탐지 결과도 같은 MaskBox 구조로 저장할 수 있어야 한다.
완료 기준
각 페이지별 마스킹 박스 목록을 저장할 수 있다.
특정 페이지의 박스만 추가·삭제할 수 있다.
후속 OCR/정규식 탐지 결과를 동일 구조로 확장할 수 있다.
Version 4. 수동 마스킹 박스 생성 기능 구현
작업 내용

사용자가 페이지 위에서 마우스로 드래그하여 마스킹 박스를 생성할 수 있도록 한다.

처리 흐름은 다음과 같다.

페이지 wrapper에서 mouseDown
→ 시작 좌표 저장
→ mouseMove 중 임시 박스 표시
→ mouseUp 시 최종 박스 생성
→ 해당 페이지 masks 배열에 저장
구현 원칙
마스킹 박스는 검은색 또는 반투명 검은색으로 표시한다.
최소 크기 이하의 드래그는 무시한다.
박스 좌표는 해당 페이지 기준 상대 좌표로 저장한다.
페이지 밖으로 벗어난 좌표는 보정한다.
처음에는 이동·리사이즈보다 생성·삭제를 우선한다.
완료 기준
각 페이지에서 드래그로 박스를 만들 수 있다.
생성된 박스가 화면에 표시된다.
페이지별로 박스가 독립 관리된다.
새 PDF를 업로드하면 기존 박스 상태가 초기화된다.
Version 5. 마스킹 박스 삭제 기능 구현
작업 내용

사용자가 잘못 생성한 마스킹 박스를 삭제할 수 있도록 한다.

구현 방식은 다음 중 하나로 시작한다.

박스 클릭 후 삭제 버튼 표시
박스 우측 상단에 X 버튼 표시
선택된 박스를 키보드 Delete로 삭제

초기 MVP에서는 X 버튼 방식이 가장 단순하다.

완료 기준
생성된 박스를 개별 삭제할 수 있다.
삭제 후 해당 페이지의 상태값에서 박스가 제거된다.
삭제된 박스는 PDF 생성 결과에도 반영되지 않는다.
Version 6. 이미지 기반 PDF 재생성 기능 구현
작업 내용

현재 렌더링된 각 페이지 canvas 또는 이미지 데이터를 기반으로 새 PDF를 생성한다.

처리 흐름은 다음과 같다.

각 페이지 원본 렌더링 canvas 확보
→ 해당 페이지 masks 좌표에 검은 박스 그리기
→ canvas를 PNG 또는 JPEG로 변환
→ pdf-lib로 새 PDF 생성
→ 페이지별 이미지 삽입
→ PDF bytes 생성
구현 원칙
결과 PDF는 이미지 기반 PDF여야 한다.
원본 PDF의 텍스트 레이어는 유지하지 않는다.
마스킹 박스는 최종 이미지에 실제로 그려져야 한다.
미리보기용 canvas를 직접 훼손하지 않도록 복사 canvas를 사용한다.
페이지 크기는 렌더링된 canvas 비율을 유지한다.
완료 기준
다운로드된 PDF에서 마스킹 영역이 실제로 검게 가려진다.
PDF 텍스트 선택 또는 복사로 원본 텍스트가 노출되지 않는다.
모든 페이지가 누락 없이 포함된다.
마스킹 박스가 없는 페이지도 정상 포함된다.
Version 7. 다운로드 링크 제공 및 메모리 정리
작업 내용

생성된 PDF bytes를 Blob으로 변환하고, URL.createObjectURL()을 사용하여 다운로드 링크를 제공한다.

처리 흐름은 다음과 같다.

pdfBytes 생성
→ Blob 생성
→ Object URL 생성
→ 다운로드 버튼 또는 링크 표시
→ 다운로드 완료 또는 새 작업 시작 시 Object URL 해제
구현 원칙
결과 PDF는 서버에 업로드하지 않는다.
다운로드 링크는 브라우저 세션성 링크로 제공한다.
새 파일 업로드 시 기존 다운로드 URL은 해제한다.
브라우저 창을 닫으면 결과물은 유지되지 않는다.
완료 기준
사용자가 masked.pdf 또는 지정된 파일명으로 PDF를 다운로드할 수 있다.
새 작업 시작 시 기존 다운로드 링크가 정리된다.
서버 저장 없이 다운로드가 가능하다.
Version 8. 정규식 기반 자동 탐지 준비
작업 내용

수동 마스킹 MVP 완성 후, 정규식 기반 자동 탐지 기능을 추가할 수 있도록 구조를 정리한다.

향후 대상은 다음과 같다.

대상	예시	탐지 방식
주민등록번호	900101-1234567	정규식
사업자등록번호	123-45-67890	정규식
전화번호	010-1234-5678	정규식
이메일	test@example.com	정규식
계좌번호	증권사·은행별 패턴	정규식 + 룰
성명	홍길동	룰 + LLM 보조
이번 단계의 구현 여부
실제 자동 탐지는 이번 MVP에서는 구현하지 않는다.
다만 자동 탐지 결과가 MaskBox 구조로 들어올 수 있도록 데이터 구조를 열어둔다.
완료 기준
수동 생성 박스와 자동 탐지 박스가 동일 구조로 관리될 수 있다.
후속 단계에서 OCR/PDF 텍스트 추출 결과를 연결할 수 있는 구조가 준비되어 있다.

6. 위험구간
6.1 좌표계 불일치
내용

화면에 표시되는 페이지 크기와 실제 canvas 크기가 다르면 마스킹 박스가 다운로드 PDF에서 어긋날 수 있다.

위험도

높음

대응방안
좌표는 반드시 canvas 기준으로 저장한다.
화면 표시 크기와 canvas 실제 크기의 scale ratio를 관리한다.
다운로드 전 테스트 PDF로 좌표 일치 여부를 검증한다.
6.2 브라우저 메모리 과부하
내용

대용량 PDF 또는 고해상도 렌더링 시 브라우저 메모리가 급증할 수 있다.

위험도

중간~높음

대응방안
초기 scale은 2.0으로 제한한다.
대용량 PDF 경고 메시지를 둔다.
필요 시 페이지별 순차 처리로 변경한다.
후속 단계에서 서버 처리 옵션을 검토한다.
6.3 PDF 렌더링 품질 저하
내용

scale이 낮으면 결과 PDF의 품질이 떨어지고, scale이 높으면 처리 속도와 메모리 부담이 커진다.

위험도

중간

대응방안
기본 scale을 정한다.
품질 우선 모드와 속도 우선 모드를 후속 옵션으로 검토한다.
테스트 문서 기준으로 출력 품질을 비교한다.
6.4 마스킹 불완전
내용

사용자가 지정한 박스가 개인정보 전체를 덮지 못하거나, 좌표 오차로 일부 글자가 남을 수 있다.

위험도

높음

대응방안
박스 생성 시 약간의 여백을 포함한다.
다운로드 전 미리보기 기능을 제공한다.
검수 후 다운로드를 기본 흐름으로 둔다.
6.5 텍스트 레이어 잔존
내용

결과 PDF가 이미지 기반이 아니라 원본 텍스트 레이어를 유지하면 마스킹 후에도 복사·검색으로 개인정보가 노출될 수 있다.

위험도

매우 높음

대응방안
결과 PDF는 새 PDF로 생성한다.
각 페이지는 이미지로만 삽입한다.
원본 PDF 페이지를 직접 수정하지 않는다.
결과 PDF에서 텍스트 선택이 불가능한지 검증한다.
6.6 서버 저장 또는 로그 노출
내용

원본 PDF, 결과 PDF, OCR 텍스트, 파일명 등이 서버 또는 로그에 남을 수 있다.

위험도

높음

대응방안
MVP에서는 서버 업로드를 사용하지 않는다.
브라우저 메모리 내 처리로 제한한다.
로그에 파일명·텍스트·개인정보를 남기지 않는다.
외부 API 사용 전 별도 보안 검토를 거친다.
6.7 외부 API 도입 시 개인정보 전송 이슈
내용

OCR API 또는 LLM API 사용 시 PDF 이미지 또는 텍스트가 외부 사업자에게 전송될 수 있다.

위험도

높음

대응방안
초기 MVP에서는 외부 API를 사용하지 않는다.
후속 도입 전 법무·보안·개인정보 검토를 수행한다.
전송 데이터 최소화 원칙을 적용한다.
API 제공자의 데이터 보관·학습 사용 정책을 확인한다.

9. 결정 로그
Decision 1. 초기 MVP는 자동 탐지보다 수동 마스킹을 우선한다.
결정

초기 MVP에서는 OCR, LLM, 외부 API를 도입하지 않고, PDF 페이지 미리보기와 수동 마스킹 기능을 우선 구현한다.

사유
자동 탐지 기능은 좌표 체계와 검수 UI가 먼저 안정화되어야 의미가 있다.
실무상 자동 탐지 결과는 누락 가능성이 있으므로 사용자 검수가 필수다.
현재 상태가 PDF 업로드 페이지만 완성된 단계이므로, 파이프라인 구축이 우선이다.
Decision 2. 결과 PDF는 이미지 기반 PDF로 생성한다.
결정

원본 PDF를 직접 수정하지 않고, 각 페이지를 이미지로 렌더링한 후 마스킹 박스를 그려 새 PDF로 생성한다.

사유
원본 텍스트 레이어 잔존에 따른 개인정보 노출 위험을 줄이기 위함이다.
사용자가 요구한 “이미지로 인쇄”와 유사한 결과를 얻기 위함이다.
PDF 내부 구조 편집보다 구현 안정성이 높다.
Decision 3. 초기 구현은 브라우저 중심으로 한다.
결정

초기 MVP에서는 서버 저장 없이 브라우저에서 PDF 렌더링, 마스킹, PDF 재생성을 처리한다.

사유
원본 PDF와 결과 PDF를 서버에 저장하지 않는 요건에 부합한다.
개인정보 외부 전송 가능성을 최소화할 수 있다.
현재 Vite/React 기반 구현 흐름에 적합하다.
Decision 4. OCR은 후속 단계로 미룬다.
결정

스캔본 PDF 자동 탐지를 위한 OCR은 수동 마스킹 MVP 완성 이후 도입 여부를 검토한다.

사유
OCR은 정확도, 성능, 좌표 품질, 한글 인식률 검증이 필요하다.
외부 OCR API 사용 시 개인정보 전송 문제가 발생할 수 있다.
수동 마스킹 기능만으로도 1차 실무 MVP가 가능하다.
Decision 5. LLM은 좌표 산정이 아니라 판단 보조 용도로만 사용한다.
결정

향후 LLM을 도입하더라도, LLM은 성명·주소 등 비정형 개인정보 여부 판단에만 사용하고, 마스킹 좌표는 PDF 텍스트 추출 또는 OCR 결과의 bounding box를 사용한다.

사유
LLM은 정확한 좌표 계산에 적합하지 않다.
좌표는 렌더링 엔진 또는 OCR 엔진의 결과값을 사용하는 것이 안정적이다.
개인정보 전송 최소화를 위해 LLM 입력은 후보 텍스트와 제한된 문맥으로 축소해야 한다.
Decision 6. 자동 탐지 결과도 수동 마스킹 박스와 동일한 데이터 구조를 사용한다.
결정

향후 정규식, OCR, LLM 기반 탐지 결과는 모두 MaskBox 구조로 변환하여 관리한다.

사유
수동 박스와 자동 박스를 동일하게 렌더링·삭제·수정할 수 있다.
다운로드 PDF 생성 로직을 단일화할 수 있다.
기능 확장 시 구조 변경 부담을 줄일 수 있다.
