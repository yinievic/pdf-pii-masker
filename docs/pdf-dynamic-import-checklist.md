# PDF Dynamic Import 수동 검증 체크리스트

Step 14A는 PDF.js와 `pdf-lib`를 초기 번들에서 분리하고 실제 사용 시점에 로드하는 성능 최적화 단계다.

## 빌드 확인

- `cd apps/web && npm run build`를 실행한다.
- 초기 `index-*.js` chunk가 PDF 최적화 전보다 유의미하게 작아졌는지 확인한다.
- `pdf-*.js`, `maskedPdfEngine-*.js`, `pdf.worker-*.js` 같은 PDF 관련 chunk가 별도로 생성되는지 확인한다.
- large chunk 경고가 다시 발생하면 어떤 chunk가 500KB를 초과하는지 기록한다.

## PDF 렌더링 확인

- 웹앱을 실행한다.
- PDF 파일을 업로드한다.
- 업로드 직후 `PDF 파일을 읽고 미리보기를 생성하는 중입니다.` 메시지가 표시되는지 확인한다.
- PDF.js와 worker 로드 실패 시 `PDF 렌더러를 준비할 수 없습니다.` 오류가 사용자에게 표시되는지 확인한다.
- `PDF 파일 페이지 확인` 후 각 PDF의 첫 페이지가 정상 표시되는지 확인한다.
- 페이지 이동 버튼과 페이지 번호 입력이 기존처럼 동작하는지 확인한다.

## 마스킹 및 다운로드 확인

- 수동 마스킹 박스를 하나 이상 만든다.
- 자동 탐지를 사용하는 경우 유지 후보와 삭제 후보가 기존처럼 `finalMasks`에 반영되는지 확인한다.
- `마스킹된 PDF 파일 다운로드` 버튼을 클릭한다.
- 클릭 시 `PDF 생성 모듈을 불러오는 중입니다.` 메시지가 표시될 수 있는지 확인한다.
- 다운로드 파일명이 기존 규칙과 동일한지 확인한다.
- 생성 후 세션성 다운로드 링크가 화면에 표시되는지 확인한다.
- 새 다운로드 생성, 파일 추가, 파일 제거, 링크 정리 클릭 시 기존 Object URL이 정리되는지 코드 경로와 브라우저 동작을 확인한다.

## 회귀 확인

- `finalMasks` 산정 방식은 변경하지 않았으므로 자동 후보 유지, 자동 후보 삭제, 수동 보완, OCR 결과 전체 폐기 흐름을 각각 확인한다.
- 다운로드 결과 PDF가 이미지 기반이며 텍스트 선택이 되지 않는지 확인한다.
- 여러 PDF를 업로드한 상태에서 각 파일의 같은 페이지 번호 마스크가 서로 섞이지 않는지 확인한다.
