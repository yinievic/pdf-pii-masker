## 1. 목표

P2의 목표는 사용자가 업로드한 PDF를 브라우저에서 페이지별로 확인하고, 수동 또는 자동 방식으로 개인(신용)정보를 마스킹한 뒤, 마스킹이 실제 반영된 이미지 기반 PDF를 다운로드할 수 있도록 하는 것이다.

최종 시스템은 다음 사용 흐름을 지원해야 한다.

1. 사용자는 PDF를 업로드하고 페이지를 확인한다.
2. 사용자는 페이지 확인 후 자동 개인정보 탐지를 실행하거나 생략할 수 있다.
2-1. 기본 자동 탐지 이후 누락 우려가 있으면 `누락 탐지 보완`을 실행해 PSM 11 기반 보완 탐지를 수행할 수 있다.
3. 자동 개인정보 탐지를 실행한 경우, 내부 OCR 서버 또는 향후 확장 가능한 OCR Provider를 통해 스캔본 PDF에서 텍스트와 좌표를 탐지한다.
4. OCR 결과는 즉시 최종 확정하지 않고, 사용자가 검토할 수 있는 자동 마스킹 후보로 표시한다.
5. 사용자는 자동 후보가 표시된 페이지를 보면서 필요한 후보는 유지하고, 불필요한 후보는 삭제·거절할 수 있다.
6. 사용자는 같은 페이지에서 수동 마스킹을 추가할 수 있다.
7. 사용자는 선택된 자동 마스킹 후보와 수동 마스킹 박스를 합산한 결과를 다운로드한다.
8. 자동 탐지를 실행하지 않은 경우에는 수동 마스킹 박스만 적용하여 다운로드한다.
9. 자동 후보가 부적절한 경우에는 파일 단위 또는 개별 후보를 해제하고 원본 PDF 기준 수동 마스킹만 수행할 수 있다.
10. 최종 다운로드 PDF는 원본 텍스트 레이어가 남지 않는 이미지 기반 PDF로 생성한다.
11. 원본 PDF와 결과 PDF는 원칙적으로 서버에 영구 저장하지 않는다.
12. OCR 서버 처리 도입 시에도 원본 파일, 페이지 이미지, OCR 텍스트, 결과물의 저장·로그 정책을 별도로 통제한다.

핵심 목표는 다음과 같다.

1. 원본 PDF 상태를 항상 보존한다.
2. 수동 마스킹과 자동 마스킹 결과를 동일한 `MaskBox` 구조로 관리한다.
3. OCR 결과는 `candidate`, `accepted`, `rejected` 상태로 검토 가능하게 한다.
4. 다운로드 로직은 수동/자동 여부와 무관하게 최종 선택된 `MaskBox[]`를 받아 처리하는 공통 함수로 통합한다.
5. 스캔본 PDF가 주 대상임을 전제로, 내부 OCR 서버 기반 확장 구조를 포함한다.

### Step 2. PDF.js 도입 및 페이지별 렌더링 구현

#### 작업 내용

업로드된 PDF를 PDF.js로 로드하고, 전체 페이지를 canvas로 렌더링한다.

처리 흐름은 다음과 같다.

```text
PDF 파일 선택
→ file.arrayBuffer()
→ PDF.js getDocument()
→ 전체 페이지 수 확인
→ 각 페이지 getPage()
→ viewport 생성
→ canvas 렌더링
→ 화면에 페이지별 미리보기 표시
```

#### 구현 원칙

- 서버 업로드 없이 브라우저에서 PDF를 렌더링한다.
- 각 페이지는 wrapper div 안에 배치한다.
- wrapper div는 향후 마스킹 박스를 overlay할 수 있는 구조로 만든다.
- 업로드 직후 PDF를 렌더링하되, 미리보기 영역은 `PDF 파일 페이지 확인` 버튼 클릭 후 표시한다.
- 파일별로 하나의 미리보기 frame을 생성한다.
- 각 문서는 한 번에 한 페이지씩 표시하고, 좌우 이동 버튼과 현재 페이지 숫자 입력으로 페이지를 전환한다.
- 미리보기 상단에는 파일명과 전체 페이지 수를 같은 줄에 표시한다.
- 렌더링 scale은 우선 2.0 기준으로 시작한다.

#### 완료 기준

- PDF 업로드 후 전체 페이지 수가 표시된다.
- 모든 페이지를 원본 순서대로 페이지 이동 UI에서 확인할 수 있다.
- 각 페이지가 이미지처럼 확인 가능하다.
- 페이지별 wrapper 구조가 생성되어 있다.
- 실제 표시 가능한 페이지가 있는 경우에만 `PDF 파일 페이지 확인` / `PDF 파일 마스킹` 버튼이 분할 표시된다.

#### 현재 상태

완료. 현재 구현은 `PDF 파일 페이지 확인` 버튼 클릭 후 파일별 미리보기 frame을 표시하며, 각 frame에서는 한 번에 한 페이지씩 확인한다. 좌우 페이지 이동 버튼과 현재 페이지 숫자 입력으로 이동할 수 있다.

### Step 3. 페이지별 상태 데이터 구조 설계

#### 작업 내용

각 페이지에 대한 렌더링 정보와 마스킹 박스 정보를 저장할 상태 구조를 설계한다.

기본 구조는 다음과 같다.

```ts
type MaskSource = "manual" | "ocr" | "regex" | "llm";

type MaskBox = {
  id: string;
  pageNumber: number;
  x: number;
  y: number;
  width: number;
  height: number;
  source?: MaskSource;
  label?: string;
};

type PageRenderState = {
  pageNumber: number;
  width: number;
  height: number;
  scale: number;
  canvasDataUrl: string;
  masks: MaskBox[];
};
```

현재 구현에서는 PDF.js 렌더링이 완료된 페이지를 대상으로 하므로 `canvasDataUrl`을 선택값이 아니라 필수값으로 둔다. `source`와 `label`은 후속 OCR/정규식/LLM 탐지 결과를 같은 구조로 수용하기 위한 선택 필드이며, `status`, `confidence`, `text`는 Step 4에서 보완한다.

#### 설계 원칙

- 마스킹 좌표는 페이지별 canvas 기준 좌표로 저장한다.
- 화면 표시 좌표와 실제 canvas 좌표의 변환 관계를 명확히 한다.
- 다운로드 시 동일 좌표를 canvas에 반영할 수 있어야 한다.
- 후속 자동 탐지 결과도 같은 `MaskBox` 구조로 저장할 수 있어야 한다.
- 화면 표시 좌표와 canvas 좌표 변환은 `displayRectToCanvasRect`, `canvasRectToDisplayRect` 같은 별도 함수로 분리한다.

#### 완료 기준

- 각 페이지별 마스킹 박스 목록을 저장할 수 있다.
- 특정 페이지의 박스만 추가·삭제할 수 있다.
- 후속 OCR/정규식 탐지 결과를 동일 구조로 확장할 수 있다.
- 특정 페이지에만 mask를 추가하는 `addMask` 함수가 준비된다.
- 특정 페이지에서 특정 mask만 삭제하는 `removeMask` 함수가 준비된다.

#### 현재 상태

완료. `PageRenderState`, `MaskBox`, canvas 기준 좌표 변환 함수, `addMask`, `removeMask`가 구현되어 있다. 실제 드래그 UI는 Step 5에서 구현 완료했다.

### Step 4. 공통 MaskBox 및 워크플로우 상태 구조 보완

#### 작업 내용

수동 마스킹과 자동 OCR 마스킹을 모두 수용할 수 있도록 `MaskBox`와 전체 워크플로우 상태를 확장한다.

현재 Step 3 구현에는 `source?`, `label?` 선택 필드가 이미 반영되어 있다. Step 4에서는 자동 후보 검토와 다운로드 대상 산정을 위해 `status`, `confidence`, `text` 등 자동 마스킹용 필드와 전체 워크플로우 상태를 보완한다.

기존 `MaskBox`는 다음과 같이 확장한다.

```ts
type MaskSource = "manual" | "ocr" | "regex" | "llm";

type MaskStatus = "candidate" | "accepted" | "rejected";

type MaskBox = {
  id: string;
  pageNumber: number;
  x: number;
  y: number;
  width: number;
  height: number;
  source: MaskSource;
  status: MaskStatus;
  label?: string;
  confidence?: number;
  text?: string;
};
```

워크플로우 상태는 다음 개념을 포함한다.

```ts
type MaskingMode =
  | "idle"
  | "manual"
  | "autoProcessing"
  | "autoReview"
  | "autoEdit"
  | "manualFromOriginal";

type WorkingBase = "original" | "autoResult";

type MaskingWorkflowState = {
  mode: MaskingMode;
  workingBase: WorkingBase;
  masks: MaskBox[];
};
```

#### 구현 원칙

- 원본 PDF는 항상 보존한다.
- OCR 결과는 즉시 확정하지 않고 기본적으로 `candidate`로 둔다.
- 사용자가 승인한 OCR 결과만 최종 다운로드 대상에 포함할 수 있어야 한다.
- OCR/정규식 후보를 해제하면 `source: "ocr"` 또는 `source: "regex"`인 박스가 최종 다운로드 대상에서 제외되어야 한다.
- 수동 박스와 자동 박스는 동일한 렌더링·삭제·다운로드 로직을 사용한다.

#### 완료 기준

- `MaskBox`가 수동/자동 탐지 결과를 모두 표현할 수 있다.
- 현재 작업 모드가 상태값으로 관리된다.
- 최종 다운로드 대상 `MaskBox[]`를 모드별로 산출할 수 있다.
- OCR/정규식 후보 해제 후에도 원본 기준 수동 마스킹을 계속할 수 있는 상태 구조가 준비된다.

### Step 5. 원본 기준 수동 마스킹 박스 생성 기능 구현

#### 작업 내용

사용자가 PDF 페이지 위에서 마우스로 드래그하여 수동 마스킹 박스를 생성할 수 있도록 한다.

처리 흐름은 다음과 같다.

```text
페이지 wrapper에서 mouseDown
→ 시작 좌표 저장
→ mouseMove 중 임시 박스 표시
→ mouseUp 시 최종 박스 생성
→ source: "manual", status: "accepted"인 MaskBox 생성
→ 해당 페이지 masks 배열 또는 공통 masks 배열에 저장
```

#### 구현 원칙

- 마스킹 박스는 검은색 또는 반투명 검은색으로 표시한다.
- 최소 크기 이하의 드래그는 무시한다.
- 박스 좌표는 해당 페이지의 실제 canvas 기준 좌표로 저장한다.
- 페이지 밖으로 벗어난 좌표는 보정한다.
- 처음에는 이동·리사이즈보다 생성·삭제를 우선한다.
- 수동 마스킹은 독립 기능이면서, 향후 OCR 결과 보완 기능으로도 재사용되어야 한다.

#### 완료 기준

- 각 페이지에서 드래그로 박스를 만들 수 있다.
- 생성된 박스가 화면에 표시된다.
- 페이지별로 박스가 독립 관리된다.
- 수동 생성 박스는 `source: "manual"`, `status: "accepted"`로 저장된다.
- 새 PDF를 업로드하면 기존 박스 상태가 초기화된다.

#### 현재 상태

완료. PDF 페이지 위 포인터 드래그로 수동 마스킹 박스를 생성하고, canvas 기준 좌표로 저장한다. 최소 크기 이하 드래그는 무시하고 페이지 경계 안으로 좌표를 보정한다.

### Step 6. 마스킹 박스 삭제 및 상태 변경 기능 구현

#### 작업 내용

사용자가 잘못 생성한 마스킹 박스를 삭제하거나 자동 탐지 후보를 거절할 수 있도록 한다.

초기 구현 방식은 다음과 같다.

1. 박스 우측 상단에 X 버튼 표시
2. 수동 박스는 삭제 시 상태에서 제거
3. 자동 후보 박스는 삭제 또는 `status: "rejected"` 처리
4. 다운로드 대상 산정 시 `rejected` 박스 제외

#### 구현 원칙

- 수동 박스와 자동 박스 모두 동일 UI에서 삭제 가능해야 한다.
- 자동 박스는 삭제 대신 rejected 상태로 남길지, 실제 배열에서 제거할지 정책을 선택할 수 있어야 한다.
- 초기 MVP에서는 UI 단순성을 위해 화면에서는 제거된 것처럼 표시한다.
- 다운로드 대상은 `status: "accepted"`인 박스만 기본 포함한다.

#### 완료 기준

- 생성된 박스를 개별 삭제할 수 있다.
- 삭제 후 해당 페이지의 상태값에서 박스가 제거되거나 rejected 처리된다.
- 삭제된 박스는 PDF 생성 결과에 반영되지 않는다.
- 자동 탐지 후보와 수동 박스 모두 동일한 삭제 흐름을 사용할 수 있다.

#### 현재 상태

완료. 수동 박스는 삭제 시 제거하고, 자동 후보는 `rejected`로 전환해 화면과 다운로드 대상에서 제외한다. 자동 검토 목록 또는 PDF overlay의 X를 다시 누르면 `review` 상태로 원복할 수 있다.

### Step 7. 수동 마스킹 결과 다운로드 구현

#### 작업 내용

현재 선택된 수동 마스킹 박스를 기준으로 이미지 기반 PDF를 생성하고 다운로드한다.

처리 흐름은 다음과 같다.

```text
각 페이지 원본 렌더링 canvas 확보
→ 최종 적용 대상 masks 산정
→ 해당 페이지 masks 좌표에 검은 박스 그리기
→ canvas를 PNG 또는 JPEG로 변환
→ pdf-lib로 새 PDF 생성
→ 페이지별 이미지 삽입
→ PDF bytes 생성
→ Blob URL 생성
→ 다운로드
```

#### 구현 원칙

- 결과 PDF는 이미지 기반 PDF여야 한다.
- 원본 PDF의 텍스트 레이어는 유지하지 않는다.
- 마스킹 박스는 최종 이미지에 실제로 그려져야 한다.
- 미리보기용 canvas를 직접 훼손하지 않도록 복사 canvas를 사용한다.
- 페이지 크기는 렌더링된 canvas 비율을 유지한다.
- 이 다운로드 로직은 후속 자동 마스킹 결과에도 재사용 가능한 공통 함수로 만든다.

#### 완료 기준

- 다운로드된 PDF에서 수동 마스킹 영역이 실제로 검게 가려진다.
- PDF 텍스트 선택 또는 복사로 원본 텍스트가 노출되지 않는다.
- 모든 페이지가 누락 없이 포함된다.
- 마스킹 박스가 없는 페이지도 정상 포함된다.
- 후속 자동 마스킹 결과도 동일 함수에 `MaskBox[]`만 넘겨 다운로드할 수 있다.

#### 현재 상태

완료. `pdf-lib`와 렌더링된 페이지 이미지를 사용해 이미지 기반 PDF를 생성한다. 마스킹 색상은 검정/흰색 중 선택할 수 있으며, 색상 선택 UI는 흰색 테두리와 체크 아이콘으로 선택 상태를 표시한다. 브라우저 다운로드 차단 이슈는 Step 14에서 다시 검토한다.

### Step 8. OCR 서버/API 구조 설계

#### 작업 내용

스캔본 PDF 자동 마스킹을 위해 내부 NAS 서버 또는 컨테이너에서 OCR을 수행하는 API 구조를 설계한다.

처리 흐름은 다음과 같다.

```text
React 앱에서 자동 마스킹 선택
→ PDF 또는 페이지 이미지 전송
→ NAS OCR API 호출
→ 서버에서 PDF 페이지 이미지 변환
→ Tesseract OCR 실행
→ OCR 원시 결과 반환
→ 프론트엔드에서 MaskBox 후보로 표시
```

#### 구현 원칙

- 외부 OCR API는 사용하지 않는다.
- 우선 내부 서버 OCR Provider를 기본값으로 설계한다.
- 향후 외부 OCR API 또는 다른 OCR 엔진으로 교체할 수 있도록 Provider 인터페이스를 분리한다.
- OCR 처리 중 원본 PDF, 페이지 이미지, OCR 텍스트가 로그에 남지 않도록 한다.
- 서버 임시파일은 처리 종료 후 삭제하는 구조로 한다.

예상 Provider 구조는 다음과 같다.

```ts
type OcrProvider = "tesseract-local" | "external-api";

type OcrRequest = {
  fileId?: string;
  pages?: number[];
  provider: OcrProvider;
};

type OcrWord = {
  pageNumber: number;
  text: string;
  x: number;
  y: number;
  width: number;
  height: number;
  confidence: number;
};
```

#### 완료 기준

- OCR API 입력·출력 구조가 정의된다.
- OCR Provider 추상화 방향이 정리된다.
- 내부 Tesseract OCR을 기본 구현 대상으로 확정한다.
- 외부 API 사용은 별도 승인 대상으로 분리된다.

### Step 9. PDF 페이지 이미지 변환 및 Tesseract OCR 실행 구현

#### 작업 내용

NAS 서버 또는 컨테이너에서 PDF 페이지를 이미지로 변환하고, Tesseract OCR을 `kor+eng` 기준으로 실행한다.

처리 흐름은 다음과 같다.

```text
PDF 수신
→ 페이지별 이미지 변환
→ tesseract page.png stdout -l kor+eng tsv 실행
→ TSV 결과 수신
→ 단어별 텍스트·좌표·신뢰도 추출
```

#### 구현 원칙

- 좌표가 필요하므로 단순 텍스트 출력이 아니라 TSV 또는 hOCR 결과를 사용한다.
- 한글과 영문이 혼재될 수 있으므로 `kor+eng`을 기본 언어로 한다.
- OCR 결과의 좌표는 이미지 좌표 기준이므로, 프론트엔드 canvas 좌표와 변환 관계를 명확히 한다.
- 서버 임시파일은 처리 종료 후 삭제한다.
- OCR 처리 실패 시 사용자에게 실패 사유를 일반화하여 표시한다.
- OCR API 컨테이너에는 `tesseract-ocr`, `tesseract-ocr-kor`, `tesseract-ocr-eng`, `poppler-utils`를 설치한다.
- OCR 실행 전 `tesseract --version`, `tesseract --list-langs`, `pdftoppm -h`로 도구 설치 상태를 검증한다.

#### 완료 기준

- 스캔본 PDF 1페이지에 대해 OCR 텍스트를 추출할 수 있다.
- 단어별 좌표와 신뢰도를 확보할 수 있다.
- 다중 페이지 PDF에서 pageNumber가 유지된다.
- OCR 결과가 프론트엔드에 전달 가능한 JSON 구조로 변환된다.

#### 현재 상태

완료. Dockerfile/docker-compose 기반 OCR API 컨테이너를 구성했다. `tesseract-ocr`, `tesseract-ocr-kor`, `tesseract-ocr-eng`, `poppler-utils`를 설치하고, NAS 호스트에서 `docker compose build/up`, `npm run verify:tools`, `/health`, 실제 `POST /ocr` 응답 검증을 완료했다. 개발 중 8080 외부 노출 없이 5173 Vite proxy를 사용한다.

### Step 10. 정규식 기반 개인정보 탐지 및 MaskBoxCandidate 생성

#### 작업 내용

OCR 원시 결과에서 정규식 기반으로 개인정보를 탐지하고, 탐지 결과와 실제 마스킹 후보를 분리해 생성한다.

탐지 규칙은 별도 파일로 관리한다. 사용자가 후속 단계에서 이 파일만 수정해 탐지 범위를 변경할 수 있도록 한다.

초기 탐지 대상은 다음 두 가지로 제한한다.

| 대상 | 예시 | 탐지 방식 |
|---|---|---|
| 주민등록번호 | 900101-1234567 | 정규식 |
| 주소 | 서울 서초구 XXXXXX | 정규식 |

#### 구현 원칙

- `Detection`은 무엇을 찾았는지 표현한다.
- `MaskBoxCandidate`는 실제로 어디를 가릴지 표현한다.
- OCR API 응답에서 `Detection[]`과 `MaskBoxCandidate[]`를 생성한다.
- Step 10에서는 UI 연결을 하지 않는다. 프론트엔드 `MaskBox` 변환과 사용자 승인·수정·삭제 흐름은 Step 11 이후에 연결한다.
- 모든 후보는 자동 확정하지 않고 `status: "review"`로 생성한다.
- 주민등록번호는 13자리 전체를 `Detection.rawText`로 보존하되, 기본 `MaskBoxCandidate`는 뒤 7자리 영역만 생성한다.
- OCR 단어 bbox가 주민등록번호 전체 하나로만 제공되는 경우 문자 인덱스 비율 기반으로 뒤 7자리 영역을 추정한다.
- 주소는 전체 주소를 `Detection.rawText`로 보존하되, `시`, `구`, `군` 이후 텍스트만 `MaskBoxCandidate`로 생성한다.
- 주소 OCR 결과가 여러 단어로 나뉘는 경우 같은 줄의 연속 단어를 합쳐 주소 후보를 만들고, 마스킹 구간에 해당하는 단어 bbox들을 합산해 후보 좌표를 만든다.
- 주소가 가까운 다음 줄까지 이어지는 경우 기존 주소 탐지 원칙은 유지하면서 다음 줄을 별도 `MaskBoxCandidate`로 생성하고 동일 `detectionId`/`groupId`로 연결한다.
- 탐지된 텍스트의 bounding box는 관련 단어들의 좌표를 합산하여 산정한다.

#### 완료 기준

- 탐지 규칙 파일에서 주민등록번호와 주소 규칙을 분리 관리할 수 있다.
- OCR 결과에서 주민등록번호와 주소를 정규식으로 탐지할 수 있다.
- OCR API 응답에 `Detection[]`이 생성된다.
- OCR API 응답에 `MaskBoxCandidate[]`가 생성된다.
- 주민등록번호 후보는 뒤 7자리 영역만 대상으로 생성된다.
- 주소 후보는 `시`, `구`, `군` 이후 영역만 대상으로 생성된다.
- 모든 후보는 `review` 상태로 생성된다.

#### 현재 상태

완료. `apps/ocr-api/src/detection/detectionRules.json`에서 주민등록번호와 주소 규칙을 분리 관리한다. `Detection`과 `MaskBoxCandidate`를 분리 생성하며, 주민등록번호는 뒤 7자리 후보만 생성한다. 주소는 기존 정규식 기반 원칙을 유지하고 줄바꿈 연속 후보는 동일 `detectionId`/`groupId`로 묶되 줄별 후보를 분리 생성한다.

### Step 11. 자동 마스킹 결과 검토 UI 구현

#### 작업 내용

자동 OCR/정규식 탐지 결과를 사용자에게 검토 가능한 마스킹 후보로 표시한다.

사용자는 다음 작업을 할 수 있어야 한다.

1. OCR 결과 전체 확인
2. 자동 후보 박스 개별 삭제 또는 원복
3. 파일 단위 자동 후보 전체 설정/해제
4. 누락된 영역 수동 추가
5. 자동 후보와 수동 마스킹을 같은 파일 단위 검토 패널에서 확인
6. OCR 결과를 유지한 상태에서 수동 보완 진행

#### 구현 원칙

- 자동 후보는 수동 박스와 시각적으로 구분할 수 있어야 한다.

- 자동 후보가 많아도 화면이 과도하게 복잡해지지 않도록 해야 한다.
- 자동 후보는 승인/폐기 버튼보다 선택/해제 상태로 관리한다.
- 사용자는 Step 15의 `현재 마스킹 상황` 패널에서 파일별 자동/수동 마스킹 상태를 확인한다.
- OCR 결과가 불완전할 수 있다는 점을 UI상 명확히 한다.

#### 완료 기준

- 자동 탐지 후보가 페이지 위에 표시된다.
- 사용자는 후보를 개별 삭제할 수 있다.
- 사용자는 후보를 그대로 다운로드 대상으로 사용할 수 있다.
- 사용자는 후보에 수동 마스킹을 추가할 수 있다.
- 사용자는 파일 단위 체크박스 또는 개별 X 버튼으로 자동 후보를 다운로드 대상에서 제외할 수 있다.

#### 현재 상태

완료. `자동 개인정보 탐지` 버튼이 OCR API를 호출하고 파일 단위 진행률 `(n/n)`을 표시한다. 자동 후보는 PDF 위에 표시되고, 검토 목록은 `MaskBoxCandidate` 단위로 표시된다. 후보 X 버튼은 삭제/원복 토글로 동작한다. 파일 단위 체크박스로 자동 후보 전체 설정/해제를 수행하며, 삭제하지 않고 유지한 자동 후보는 `finalMasks`에 포함된다.

### Step 11A. OCR 탐지 품질 최소 개선 및 검증

5. 개선 전후 탐지 결과 비교

- 주민등록번호 탐지 전 공백 축소, 하이픈 유사 문자 통일, 전각 숫자 변환, `I/l/|/O/Q` 제한적 숫자 보정을 적용한다.

- 주민등록번호 후보는 confidence가 낮아도 `review` 상태로 생성한다.
- 주민등록번호 기본 후보는 뒤 7자리 영역만 생성한다.
- 기본 OCR과 숫자 전용 OCR 결과를 비교할 수 있다.
- whitelist 적용 결과를 비교할 수 있다.
- grayscale/thresholding 기반 전처리 결과를 비교할 수 있다.
- 정규식 탐지 실패 가능성이 있는 숫자 후보를 확장 후보로 별도 산출할 수 있다.
- API 응답 구조는 기존 `words`, `detections`, `maskBoxCandidates`를 유지한다.

#### 현재 상태

완료. 운영 탐지 로직에는 주민등록번호 normalization과 같은 줄 인접 word 결합 탐지를 반영했다. `Detection.rawText`는 OCR 원문을 보존하고 `normalizedText`를 선택 필드로 제공한다. OCR DPI 변경, 이미지 전처리, 숫자 전용 재OCR, whitelist 재OCR, OCR 엔진 교체는 후속 개선 과제로 문서화했다.

### Step 12. 자동 후보 취사선택 및 최종 적용 대상 산정

#### 작업 내용

자동 마스킹 결과를 별도의 확정 파일로 취급하지 않고, 사용자가 현재 PDF 페이지 위에서 자동 후보를 보며 필요한 후보만 유지하고 불필요한 후보는 제외한 뒤, 같은 페이지에서 수동 마스킹을 추가할 수 있도록 한다.

사용자의 실제 최종 선택 경로는 다음과 같다.

```text
1. PDF 업로드
2. 페이지 확인
3. 자동 개인정보 탐지 실행 또는 생략
4. 자동 탐지를 실행한 경우, 페이지 위 자동 후보를 보며 취사선택
5. 같은 페이지에서 수동 마스킹 추가
6. 선택된 자동 후보 + 수동 마스킹을 최종 적용 대상으로 산정
7. 이미지 기반 PDF 다운로드
```

자동 탐지를 실행하지 않은 경우에는 수동 마스킹 박스만 최종 적용 대상으로 산정한다.

#### 구현 원칙

- Step 12는 새로운 OCR 탐지 기능을 추가하는 단계가 아니라, 사용자의 선택 결과를 다운로드 직전의 최종 `MaskBox[]`로 정리하는 단계로 본다.
- 자동 후보 중 사용자가 유지한 후보는 최종 적용 대상에 포함한다.
- 사용자가 삭제 또는 거절한 자동 후보는 최종 적용 대상에서 제외한다.
- 사용자가 페이지에서 직접 추가한 수동 마스킹 박스는 최종 적용 대상에 포함한다.
- 자동 탐지를 실행하지 않은 경우에도 동일한 산정 함수를 사용하되, 수동 박스만 포함한다.
- 자동 후보 해제는 별도 다운로드 경로가 아니라, 해제된 후보를 `finalMasks`에서 제외하고 원본 기준 수동 마스킹 상태를 유지하는 기능으로 본다.
- 원본 PDF의 렌더링 상태는 항상 유지되어야 한다.

최종 적용 대상 산정의 개념은 다음과 같다.

```ts
type FinalMaskOptions = {
  includeAutoCandidates: boolean;
  includeManualMasks: boolean;
  excludeRejected: boolean;
};

const finalMasks = getFinalMasks({
  masks,
  includeAutoCandidates: true,
  includeManualMasks: true,
  excludeRejected: true,
});
```

#### 완료 기준

- 자동 탐지를 실행하지 않은 경우에도 수동 마스킹만으로 다운로드 대상 `MaskBox[]`를 산정할 수 있다.
- 자동 탐지를 실행한 경우, 사용자가 유지한 자동 후보와 수동 마스킹 박스를 함께 최종 적용 대상으로 산정할 수 있다.
- 사용자가 삭제·거절한 자동 후보는 최종 적용 대상에서 제외된다.
- 자동 후보 해제 시 해당 후보는 최종 적용 대상에서 제외되고, 원본 기준 수동 마스킹은 유지된다.
- 다운로드 버튼 클릭 시 Step 13의 공통 다운로드 엔진에 전달할 `finalMasks`가 일관되게 생성된다.

#### 현재 상태

완료. `getFinalMasks` 산정 함수를 분리했고, 기본 옵션은 수동 accepted 박스와 삭제되지 않은 자동 후보를 포함하며 rejected 후보를 제외한다. 파일 단위 또는 개별 후보 해제 시 해당 자동 후보는 `finalMasks`에서 제외되고 원본 기준 수동 마스킹은 유지된다. 다운로드 버튼은 산정된 `finalMasks`를 단일 입력값으로 PDF 생성 경로에 전달한다.

### Step 13. 최종 다운로드 엔진 통합

#### 작업 내용

Step 12에서 산정한 최종 `MaskBox[]`를 입력값으로 받아, 수동/자동/자동 보완 여부와 관계없이 하나의 다운로드 엔진이 이미지 기반 PDF를 생성하도록 통합한다.

공통 함수의 개념은 다음과 같다.

```ts
generateMaskedPdf({
  sourcePdf,
  pages,
  masks: finalMasks,
  fileName,
});
```

모드별 입력은 다음과 같다.

| 사용자 작업 상태 | Step 13에 전달되는 masks |
|---|---|
| 자동 탐지 없이 수동 마스킹만 수행 | 수동 박스 중 최종 적용 대상 |
| 자동 탐지 후 후보를 그대로 유지 | 유지된 자동 후보 |
| 자동 후보 일부 삭제 후 수동 보완 | 유지된 자동 후보 + 수동 보완 박스 |
| 자동 후보 해제 후 수동 작업 | 해제 후보 제외, 수동 박스만 적용 |

#### 구현 원칙

- PDF 생성 알고리즘을 모드별로 새로 만들지 않는다.
- Step 7의 수동 다운로드 로직을 공통 함수로 분리·일반화한다.
- 최종 적용 대상 박스 산정 함수와 PDF 생성 함수를 분리한다.
- Step 13은 `finalMasks`만 신뢰하고, 박스의 출처가 수동인지 자동인지에 따라 별도의 PDF 생성 경로를 만들지 않는다.
- 결과 PDF는 항상 이미지 기반 PDF로 생성한다.
- 원본 텍스트 레이어는 남기지 않는다.
- 다운로드 전 적용 대상 박스 수와 페이지 수를 검증한다.

#### 완료 기준

- 모든 작업 상태에서 동일한 다운로드 엔진을 사용한다.
- Step 12에서 생성한 `finalMasks`만 결과 PDF에 반영된다.
- 결과 PDF의 페이지 수가 원본과 일치한다.
- 삭제·거절된 박스는 결과 PDF에 반영되지 않는다.
- 자동 후보와 수동 보완 박스가 함께 있는 경우에도 둘 다 정확한 위치에 반영된다.
- 결과 PDF에서 텍스트 선택·검색으로 원본 텍스트가 노출되지 않는다.

#### 현재 상태

완료. `generateMaskedPdf` 공통 엔진을 분리했고, 다운로드 버튼은 Step 12에서 산정한 `finalMasks`만 전달한다. 엔진은 페이지 렌더링 정보와 박스 좌표를 검증하고, 생성된 이미지 기반 PDF의 페이지 수가 입력 페이지 수와 일치하는지 확인한다. 여러 PDF가 동시에 업로드된 경우 다운로드 버튼은 파일별로 `generateMaskedPdf({ documents: [pdf], masks: finalMasks })`를 호출해 결과 PDF가 합쳐지지 않도록 한다. 각 페이지의 mask id와 `finalMasks`를 교차해 해당 파일 페이지의 최종 박스만 적용한다.

### Step 14. 다운로드 링크 제공 및 메모리·임시파일 정리

#### 작업 내용

생성된 PDF bytes를 Blob으로 변환하고, `URL.createObjectURL()`을 사용하여 다운로드 링크를 제공한다.

처리 흐름은 다음과 같다.

```text
pdfBytes 생성
→ Blob 생성
→ Object URL 생성
→ 다운로드 버튼 또는 링크 표시
→ 다운로드 완료 또는 새 작업 시작 시 Object URL 해제
→ OCR 서버 임시파일 삭제 확인
```

#### 구현 원칙

- 결과 PDF는 서버에 영구 저장하지 않는다.
- 다운로드 링크는 브라우저 세션성 링크로 제공한다.
- 새 파일 업로드 시 기존 다운로드 URL은 해제한다.
- 브라우저 창을 닫으면 결과물은 유지되지 않는다.
- OCR 서버를 사용하는 경우 임시파일 삭제 정책을 확인한다.

#### 완료 기준

- 사용자가 지정된 파일명으로 PDF를 다운로드할 수 있다.
- 새 작업 시작 시 기존 다운로드 링크가 정리된다.
- 브라우저 메모리 URL이 적절히 해제된다.
- OCR 임시파일이 서버에 남지 않는다.

#### 현재 상태

완료. 생성된 PDF bytes를 Blob과 Object URL로 변환하고, 단일/복수 파일 모두 자동 다운로드를 실행하지 않고 파일별 Object URL 링크 목록을 제공한다. 복수 파일은 하나의 PDF로 병합하지 않는다. 링크 클릭 시 기본 다운로드를 먼저 막고, 지원 브라우저에서는 `showSaveFilePicker`로 PDF 저장 대화상자를 연다. 기본 파일명은 원본명에 `_masked`를 붙인다. `showSaveFilePicker`를 사용할 수 없는 브라우저 또는 비보안 접속에서는 자동 다운로드로 폴백하지 않고 사용자에게 제약 안내를 표시한다. 웹 보안상 업로드 원본 파일의 실제 폴더 경로는 접근할 수 없으므로 기본 폴더는 브라우저 정책에 따른다. 새 다운로드 생성, 수동 마스킹 추가, 새 파일 업로드, 파일 제거, 컴포넌트 언마운트 시 기존 Object URL을 `URL.revokeObjectURL()`로 해제한다. OCR API는 요청별 임시 디렉터리를 사용하고 `recognizePdf`의 `finally` 블록에서 성공·실패와 무관하게 `rm(workDir, { recursive: true, force: true })`를 호출한다.

### Step 14A. PDF 라이브러리 지연 로딩 최적화

#### 작업 내용

Vite large chunk 경고를 줄이기 위해 PDF 관련 대형 라이브러리를 실제 사용 시점에만 로드한다.

#### 구현 원칙

- PDF.js와 PDF.js worker는 PDF 렌더링 함수 실행 시점에 `dynamic import`로 로드한다.
- `maskedPdfEngine`과 그 내부의 `pdf-lib`는 다운로드 버튼 클릭 시점에 `dynamic import`로 로드한다.
- 기존 Step 12의 `finalMasks` 산정 방식과 Step 13의 다운로드 입력 구조는 변경하지 않는다.
- PDF.js worker 초기화 실패는 사용자에게 보이는 오류 상태로 표시한다.
- 각 로딩 단계는 별도 loading 상태로 표시한다.

#### 완료 기준

- 앱 초기 번들에 `pdfjs-dist`와 `pdf-lib`가 직접 포함되지 않는다.
- PDF 업로드 후 렌더링 시점에 PDF.js가 로드된다.
- 다운로드 버튼 클릭 시점에 `maskedPdfEngine`과 `pdf-lib`가 로드된다.
- `npm run build`에서 large chunk 경고가 사라지거나 초기 chunk 크기가 유의미하게 감소한다.
- 주요 수동 검증 체크리스트가 문서화된다.

#### 현재 상태

완료. `App.tsx`의 PDF.js 및 worker 정적 import를 제거하고 `renderPdfPages` 실행 시점에 로드하도록 변경했다. `maskedPdfEngine` 정적 import도 제거하고 다운로드 버튼 클릭 시점에 로드하도록 변경했다. 빌드 결과 초기 `index` chunk는 약 958KB에서 약 163KB로 감소했고, PDF 관련 코드는 별도 chunk로 분리됐다.

### Step 14B. 자동/수동 마스킹 검토 UI 정리

#### 작업 내용

Step 15의 `현재 마스킹 상황` 패널로 이어지도록, 자동 후보와 수동 마스킹 상태를 파일 단위로 더 명확하게 검토할 수 있도록 UI를 정리한다.

#### 구현 원칙

- 다운로드 버튼에는 전체 `finalMasks` 합산 개수를 표시하지 않는다.
- 자동 탐지 결과 패널은 파일별로 자동 탐지 열과 수동 마스킹 열을 나누어 표시한다.
- 자동 탐지 열과 수동 마스킹 열의 폭은 대략 2:1 비율로 둔다.
- 두 열 사이에는 얇은 세로 구분선을 둔다.
- 파일 단위 자동 후보 전체 설정/해제는 자동 탐지 열 제목 행 우측 체크박스로 제공한다.
- 수동 마스킹 열에는 파일별 수동 마스킹 개수, 페이지 위치, 삭제 버튼을 표시한다.
- 자동 후보 전체 승인/폐기 버튼은 제거하고, 선택/해제 상태를 기준으로 `finalMasks`를 산정한다.
- 마스킹 색상 기본값은 흰색으로 두고, 색상 버튼 순서는 흰색, 검정 순서로 둔다.

#### 완료 기준

- 다운로드 버튼 문구에서 `finalMasks` 합산 개수가 제거된다.
- 파일별 자동 후보를 체크박스로 전체 설정/해제할 수 있다.
- 자동 탐지 열과 수동 마스킹 열이 분리되어 표시된다.
- 수동 마스킹 열에서 파일별 수동 마스킹 개수와 페이지 위치를 확인하고 삭제할 수 있다.
- 기본 마스킹 색상이 흰색으로 적용된다.

#### 현재 상태

완료. 다운로드 버튼의 개수 표기를 제거했고, 자동 탐지 결과 패널을 파일별 자동/수동 2열 구조로 변경했다. 파일 단위 체크박스는 자동 탐지 제목 행 우측에 배치했으며, 수동 마스킹 열에는 파일별 수동 마스킹 개수와 페이지 위치 및 삭제 버튼을 표시한다. 마스킹 색상 기본값은 흰색이다.

### Step 15. 사용자 확인 및 안전장치 구현

#### 작업 내용

최종 다운로드 전 별도의 총합 확인 박스를 두지 않고, 파일별 검토가 가능한 `현재 마스킹 상황` 패널을 안전장치의 중심으로 사용한다.

포함 기능은 다음과 같다.

1. 기존 `자동 탐지 결과` 패널명을 `현재 마스킹 상황`으로 변경
2. 마스킹 색상 선택을 `현재 마스킹 상황` 제목 행 우측으로 이동
3. 파일별 자동 탐지 상태 표시
4. 자동 탐지 미실행 파일은 안내 문구로 표시하고 제목 옆 상태값은 생략
5. 자동 탐지 실행 후에는 파일별 `선택됨/해제됨` 수량 표시
6. 파일별 수동 마스킹 수량과 페이지 위치 표시
7. 새로고침 또는 파일 교체 시 작업 손실 경고

#### 구현 원칙

- 전체 파일의 마스킹 총합은 사용자가 판단하기 어려운 값이므로 별도 다운로드 전 확인 박스로 표시하지 않는다.
- 자동/수동 마스킹 상태는 파일별로 확인할 수 있어야 한다.
- 자동 탐지를 실행하지 않아도 현재 파일의 수동 마스킹 상황은 확인할 수 있어야 한다.
- 다운로드 버튼 문구와 다운로드 경로는 Step 13~14의 `finalMasks` 산정과 파일별 PDF 생성 흐름을 유지한다.

#### 완료 기준

- `현재 마스킹 상황` 패널이 PDF 페이지 확인 후 표시된다.
- 패널 제목 행 우측에서 마스킹 색상을 선택할 수 있다.
- 자동 탐지 미실행 파일은 `자동 탐지가 아직 실행되지 않았습니다.` 안내 문구로 표시된다.
- 자동 탐지 실행 후 파일별 `선택됨/해제됨` 수량이 표시된다.
- 수동 마스킹 열에서 파일별 수동 마스킹 수량과 페이지 위치를 확인하고 삭제할 수 있다.
- 새 파일 업로드, 파일 제거, 새로고침 시 작업 손실 가능성이 안내된다.

#### 현재 상태

완료. `다운로드 전 확인` 박스와 검토 확인 체크박스 의존을 제거하고, `현재 마스킹 상황` 패널을 파일별 검토 지점으로 확장했다. 마스킹 색상 선택은 패널 제목 행 우측에 배치했다. 자동 탐지를 실행하지 않은 파일도 패널에 표시되며, 자동 탐지 미실행 시에는 목록 영역에 `자동 탐지가 아직 실행되지 않았습니다.`를 표시한다. 자동 탐지 실행 후에는 파일별 선택/해제 수량을 `자동 탐지` 제목 옆에 표시한다. 다운로드 버튼은 링크 생성만 수행하며, 수동 마스킹을 추가하면 해당 파일의 기존 다운로드 링크를 제거한다.

### Step 16. 기능 개선 및 텍스트 정돈

#### 작업 내용

Step 15까지 완성된 핵심 워크플로우를 유지하면서, 실제 사용자 안내 문구와 자동 탐지 보완 흐름을 정리한다.

포함 기능은 다음과 같다.

1. `데모 흐름 보기` 버튼을 `사용 방법` 버튼으로 변경
2. 현재 기능에 맞춰 사용 방법 페이지를 재구성
3. 페이지 상단과 하단 안내 문구 정리
4. 기본 자동 탐지 이후 실행할 수 있는 `누락 탐지 보완` 기능 추가
5. `누락 탐지 보완`은 사용자 설정값으로 노출하지 않고 내부적으로 Tesseract PSM 11을 사용
6. `public/favicon.ico`를 웹앱 파비콘으로 적용

#### 구현 원칙

- 기본 OCR 동작은 기존 자동 개인정보 탐지 흐름으로 유지한다.
- `누락 탐지 보완`은 기본 자동 탐지 결과를 대체하지 않고 후보를 추가 병합한다.
- 두 버튼은 별도 버튼으로 두되, 하나의 자동 탐지 기능군처럼 보이도록 같은 그룹 안에 배치한다.
- 사용자 UI에는 PSM 값이나 OCR 엔진 설정명을 직접 노출하지 않는다.
- OCR API 응답 구조와 기존 `MaskBox` 변환 구조는 유지한다.
- 보완 탐지 요청이 실제 PSM 11로 처리됐는지 응답 메타데이터와 검증 스크립트로 확인한다.

#### 완료 기준

- 사용 방법 페이지가 현재 기능 흐름과 일치한다.
- `자동 개인정보 탐지`와 `누락 탐지 보완`이 하나의 기능군으로 시각적으로 묶인다.
- 기본 자동 탐지 전에는 `누락 탐지 보완`을 실행할 수 없다.
- 기본 자동 탐지 후 `누락 탐지 보완`을 실행하면 `/ocr?psm=11` 요청으로 후보를 추가 탐지한다.
- 보완 탐지 결과는 기존 자동 후보와 병합되고, 사용자가 같은 `현재 마스킹 상황` 패널에서 검토할 수 있다.
- 브라우저 탭에서 `public/favicon.ico`가 파비콘으로 로드된다.

#### 현재 상태

구현 및 빌드 검증 완료. 사용 방법과 상단 안내 문구를 현재 워크플로우에 맞게 정리했고, `자동 개인정보 탐지`와 `누락 탐지 보완`을 하나의 기능군으로 묶어 배치했다. 기본 자동 탐지 후 `누락 탐지 보완`을 실행하면 `/ocr?psm=11`로 보완 탐지를 요청하고 기존 후보에 병합한다. OCR API 응답에 실제 적용된 `ocrOptions.psm`을 포함하고, 프론트와 프록시 검증 스크립트에서 PSM 11 적용 여부를 확인한다. `public/favicon.ico`를 `index.html`에 연결했다. 사용자 수동 확인을 남겨 둔다.

## 6. 위험구간

### 6.1 좌표계 불일치

#### 내용

화면에 표시되는 페이지 크기, PDF.js canvas 크기, OCR 이미지 크기, 최종 다운로드 canvas 크기가 다르면 마스킹 박스가 어긋날 수 있다.

#### 위험도

매우 높음

#### 대응방안

- 좌표는 기본적으로 실제 canvas 기준으로 저장한다.
- OCR 이미지 좌표와 프론트엔드 canvas 좌표의 변환 비율을 명시적으로 관리한다.
- 화면 표시 크기와 canvas 실제 크기의 scale ratio를 별도 함수로 계산한다.
- OCR 결과를 MaskBox로 변환할 때 좌표계 변환 테스트를 수행한다.
- 다운로드 전 샘플 PDF로 좌표 일치 여부를 검증한다.

---

### 6.2 OCR 좌표 품질 문제

#### 내용

Tesseract OCR 결과의 bounding box가 실제 민감정보 전체를 정확히 덮지 못할 수 있다.

#### 위험도

높음

#### 대응방안

- 탐지된 박스에 기본 padding을 부여한다.
- 인접 단어 결합 시 전체 bounding box를 넉넉하게 산정한다.
- 자동 결과는 확정값이 아니라 후보로 표시한다.
- 사용자가 자동 후보를 확인·수정할 수 있도록 한다.

---

### 6.3 OCR 인식률 한계

#### 내용

스캔 품질, 기울어진 문서, 낮은 해상도, 복잡한 표, 필기체 등으로 인해 OCR 인식률이 낮을 수 있다.

#### 위험도

높음

#### 대응방안

- OCR 결과를 반드시 사용자 검토 대상으로 둔다.
- 신뢰도 낮은 항목은 별도 표시한다.
- 전처리 기능은 후속 단계에서 검토한다.
- 자동 탐지 실패 시 원본 기준 수동 마스킹으로 전환할 수 있게 한다.

---

### 6.4 브라우저 및 서버 성능 문제

#### 내용

대용량 PDF, 고해상도 렌더링, OCR 처리로 인해 브라우저 또는 NAS 서버 성능 부담이 커질 수 있다.

#### 위험도

중간~높음

#### 대응방안

- 초기 scale은 2.0으로 제한한다.
- 대용량 PDF 경고 메시지를 둔다.
- OCR은 페이지별 순차 처리 또는 큐 방식으로 전환 가능하게 설계한다.
- 처리 가능한 최대 페이지 수와 파일 크기 제한을 둔다.
- Vite 빌드의 대형 chunk 경고는 Step 14A에서 PDF.js와 `pdf-lib`의 `dynamic import`를 적용해 초기 chunk 크기를 줄였다.
- 추가적인 `manualChunks` 또는 `build.chunkSizeWarningLimit` 조정은 실제 초기 로딩 개선이 더 필요한지 확인한 뒤 후속 단계에서 결정한다.

---

### 6.5 텍스트 레이어 잔존

#### 내용

결과 PDF가 이미지 기반이 아니라 원본 텍스트 레이어를 유지하면 마스킹 후에도 복사·검색으로 개인정보가 노출될 수 있다.

#### 위험도

매우 높음

#### 대응방안

- 결과 PDF는 새 PDF로 생성한다.
- 각 페이지는 이미지로만 삽입한다.
- 원본 PDF 페이지를 직접 수정하지 않는다.
- 결과 PDF에서 텍스트 선택이 불가능한지 검증한다.

---

### 6.6 서버 저장 또는 로그 노출

#### 내용

OCR 서버 도입 시 원본 PDF, 페이지 이미지, OCR 텍스트, 파일명, 탐지 결과가 서버 또는 로그에 남을 수 있다.

#### 위험도

매우 높음

#### 대응방안

- 서버 임시파일은 처리 종료 후 삭제한다.
- 로그에 파일명·텍스트·개인정보를 남기지 않는다.
- OCR 결과 저장 여부는 별도 승인 전까지 금지한다.
- 개발 로그에도 민감정보가 출력되지 않도록 한다.

---

### 6.7 외부 API 도입 시 개인정보 전송 이슈

#### 내용

향후 OCR API 또는 LLM API 사용 시 PDF 이미지 또는 텍스트가 외부 사업자에게 전송될 수 있다.

#### 위험도

높음

#### 대응방안

- 초기 자동 OCR은 내부 NAS 서버 OCR을 기본값으로 한다.
- 외부 API 사용은 별도 법무·보안·개인정보 검토 후 승인한다.
- 전송 데이터 최소화 원칙을 적용한다.
- API 제공자의 데이터 보관·학습 사용 정책을 확인한다.

---

### 6.8 자동 탐지 결과에 대한 과신

#### 내용

사용자가 OCR 자동 탐지 결과를 완전한 것으로 오인하고 검토 없이 다운로드할 수 있다.

#### 위험도

높음

#### 대응방안

- 자동 탐지는 후보임을 명확히 표시한다.
- `현재 마스킹 상황` 패널에서 파일별 자동 후보 선택/해제 상태와 수동 마스킹 상태를 표시한다.
- 자동 탐지를 실행하지 않은 파일은 별도 안내 문구로 표시한다.
- 사용자가 해제한 자동 후보는 `finalMasks`에서 제외한다.

## 7. rollback 기준

### Rollback 1. PDF 렌더링 실패

#### 기준

다음 문제가 발생하면 PDF 렌더링 구현 직전 상태로 되돌린다.

- 기존 업로드 페이지가 정상 작동하지 않음
- PDF 업로드 후 화면이 멈춤
- 다중 페이지 PDF에서 렌더링 오류 발생
- 기존 UI 구조가 크게 훼손됨

#### 복구 기준

- 기존 PDF 업로드 페이지가 정상 표시되어야 한다.
- 파일 선택 기능이 정상 작동해야 한다.

---

### Rollback 2. 상태 구조 변경으로 기존 Step 1~3 기능 훼손

#### 기준

Step 4에서 MaskBox 및 Workflow 상태 구조를 확장한 후 다음 문제가 발생하면 Step 3 완료 상태로 되돌린다.

- PDF 업로드 후 페이지 상태가 생성되지 않음
- 페이지별 렌더링 결과가 표시되지 않음
- 기존 페이지별 상태 구조가 깨짐
- 새 PDF 업로드 시 상태 초기화가 되지 않음

#### 복구 기준

- Step 1~3의 업로드, 렌더링, 페이지별 상태 관리 기능이 정상이어야 한다.

---

### Rollback 3. 마스킹 박스 좌표 불일치

#### 기준

다음 문제가 발생하면 수동 마스킹 박스 기능 구현 직전 상태로 되돌린다.

- 화면상 박스 위치와 다운로드 PDF의 마스킹 위치가 다름
- 페이지 스크롤 또는 확대·축소 시 좌표가 어긋남
- 페이지별 박스가 다른 페이지에 적용됨
- OCR 좌표 변환 후 박스가 엉뚱한 위치에 표시됨

#### 복구 기준

- PDF 페이지 미리보기 기능은 정상 유지되어야 한다.
- 마스킹 기능은 비활성화하더라도 렌더링은 정상이어야 한다.

---

### Rollback 4. PDF 재생성 결과 불량

#### 기준

다음 문제가 발생하면 PDF 다운로드 기능 구현 직전 상태로 되돌린다.

- 생성된 PDF가 열리지 않음
- 페이지 일부가 누락됨
- 페이지 순서가 변경됨
- 마스킹 박스가 반영되지 않음
- 결과 PDF에 원본 텍스트 레이어가 남아 있음

#### 복구 기준

- 수동 마스킹 박스 UI는 정상 유지되어야 한다.
- PDF 생성 버튼은 임시 비활성화할 수 있다.

---

### Rollback 5. OCR 서버 처리 실패

#### 기준

다음 문제가 반복되면 자동 OCR 기능을 비활성화하고 수동 마스킹 기능만 유지한다.

- OCR 서버가 PDF를 처리하지 못함
- OCR 처리 중 서버 또는 컨테이너가 불안정해짐
- OCR 결과 좌표가 일관되게 부정확함
- OCR 처리 시간이 업무상 사용하기 어려운 수준임

#### 복구 기준

- 수동 마스킹 및 수동 다운로드 기능은 정상 유지되어야 한다.
- 자동 마스킹 버튼은 임시 비활성화할 수 있다.

---

### Rollback 6. 브라우저 또는 서버 성능 문제

#### 기준

다음 문제가 반복되면 고해상도 렌더링, 전체 페이지 동시 렌더링 또는 OCR 처리 방식을 되돌린다.

- 일반 PDF에서 브라우저가 멈춤
- 메모리 사용량이 과도함
- OCR 처리로 NAS 서버가 불안정해짐
- 페이지 렌더링 또는 OCR 시간이 업무상 사용하기 어려운 수준임

#### 복구 기준

- 렌더링 scale을 낮춘다.
- 페이지별 순차 렌더링 방식으로 전환한다.
- OCR 페이지 수 제한을 도입한다.
- 자동 OCR을 비활성화하고 수동 마스킹으로 우회한다.

---

### Rollback 7. 보안 정책 위반 가능성

#### 기준

다음 상황이 확인되면 즉시 해당 기능을 비활성화하거나 되돌린다.

- 원본 PDF가 서버에 영구 저장됨
- 결과 PDF가 서버에 영구 저장됨
- OCR 텍스트가 로그에 남음
- 개인정보가 포함된 페이지 이미지가 삭제되지 않음
- 사용자의 명시적 승인 없이 외부 API로 데이터가 전송됨

#### 복구 기준

- 브라우저 내 수동 처리만 남긴다.
- OCR 서버 기능을 비활성화한다.
- 외부 전송 코드를 제거한다.
- 로그 출력 코드를 제거한다.
- 임시파일 삭제 로직을 검증한다.

---

## 8. 결정 로그

### Decision 1. 최종 시스템은 수동/자동/자동결과수정 흐름을 모두 지원한다.

#### 결정

P2는 단순 수동 마스킹 도구가 아니라, 사용자가 수동 마스킹과 자동 OCR 마스킹을 선택하고, 자동 결과를 그대로 사용하거나 수동으로 보완하거나 폐기할 수 있는 구조로 설계한다.

#### 사유

- 실제 업무에서는 스캔본 PDF가 주 대상일 가능성이 높다.
- OCR 결과는 불완전할 수 있으므로 사용자 검수·수정 흐름이 필수다.
- 자동 결과가 만족스럽지 않은 경우 원본 기준 수동 작업으로 돌아갈 수 있어야 한다.

---

### Decision 2. 원본 PDF 상태는 항상 보존한다.

#### 결정

자동 OCR 결과와 관계없이 원본 PDF 상태를 유지하고, OCR 결과 폐기 후 원본 기준 수동 마스킹으로 전환할 수 있도록 한다.

#### 사유

- OCR 결과가 부정확한 경우 원본에서 다시 작업해야 한다.
- 자동 처리 결과를 원본처럼 취급하면 사용자 선택지가 제한된다.
- 결과물 검수 및 rollback에 유리하다.

---

### Decision 3. 수동 마스킹은 독립 기능이자 자동 OCR 결과 보정 기능으로 설계한다.

#### 결정

수동 마스킹 기능은 원본 기준 독립 기능으로 구현하되, 자동 OCR 결과에 누락된 영역을 추가하거나 잘못된 영역을 보완하는 공통 편집 기능으로 재사용한다.

#### 사유

- 수동 마스킹과 자동 결과 수정 UI를 별도로 만들면 중복이 커진다.
- OCR 결과 보완은 결국 MaskBox 추가·삭제와 동일한 문제다.
- 공통화하면 다운로드 로직도 단순해진다.

---

### Decision 4. OCR 결과는 즉시 확정하지 않고 후보 상태로 둔다.

#### 결정

OCR/정규식 탐지 결과는 기본적으로 `status: "candidate"`인 `MaskBox`로 저장한다.

#### 사유

- OCR은 인식 누락과 오탐 가능성이 있다.
- 사용자가 자동 결과를 확인·삭제·보완할 수 있어야 한다.
- 최종 다운로드 전에 사용자의 검수 흐름을 둘 수 있다.

---

### Decision 5. 결과 PDF는 이미지 기반 PDF로 생성한다.

#### 결정

원본 PDF를 직접 수정하지 않고, 각 페이지를 이미지로 렌더링한 후 마스킹 박스를 그려 새 PDF로 생성한다.

#### 사유

- 원본 텍스트 레이어 잔존에 따른 개인정보 노출 위험을 줄이기 위함이다.
- 사용자가 요구한 “이미지로 인쇄”와 유사한 결과를 얻기 위함이다.
- PDF 내부 구조 편집보다 구현 안정성이 높다.

---

### Decision 6. 자동 OCR은 내부 NAS 서버 기반 Tesseract 방식을 우선 검토한다.

#### 결정

스캔본 PDF 자동 탐지를 위해 외부 OCR API보다 내부 NAS 서버 또는 컨테이너에서 Tesseract OCR을 실행하는 구조를 우선 검토한다.

#### 사유

- 개인정보와 업무문서가 외부 서버로 전송되는 위험을 줄일 수 있다.
- 외부 API 사용은 별도 법무·보안·개인정보 검토가 필요하다.
- Tesseract TSV/hOCR 결과를 사용하면 텍스트뿐 아니라 좌표를 확보할 수 있다.

---

### Decision 7. OCR/정규식/수동 결과는 모두 동일한 MaskBox 구조를 사용한다.

#### 결정

수동 생성 박스, OCR 탐지 박스, 정규식 탐지 박스는 모두 `MaskBox` 구조로 변환하여 관리한다.

#### 사유

- 수동 박스와 자동 박스를 동일하게 렌더링·삭제·수정할 수 있다.
- 다운로드 PDF 생성 로직을 단일화할 수 있다.
- 기능 확장 시 구조 변경 부담을 줄일 수 있다.

---

### Decision 8. 다운로드 로직은 Step 12에서 산정한 finalMasks를 받는 공통 엔진으로 통합한다.

#### 결정

수동/자동/자동수정 여부에 따라 다운로드 함수를 나누지 않고, 사용자가 페이지에서 취사선택한 자동 후보와 수동 마스킹 박스를 Step 12에서 `finalMasks`로 산정한 뒤 하나의 PDF 생성 함수에 전달한다.

#### 사유

- 다운로드 로직 중복을 줄일 수 있다.
- 자동 후보 유지·해제, 수동 보완 등 분기 처리가 명확해진다.
- Step 13은 박스 출처가 아니라 최종 적용 대상 여부만 기준으로 PDF를 생성하게 되어 테스트 범위가 줄어든다.
- 사용자가 실제로 수행하는 “자동 후보 취사선택 + 수동 마스킹 + 다운로드” 흐름과 구현 구조가 일치한다.

---

### Decision 9. 외부 OCR API와 LLM API는 별도 승인 전까지 제외한다.

#### 결정

외부 OCR API 및 외부 LLM API는 초기 구현 범위에서 제외하고, 내부 OCR 구조 안정화 후 별도 승인 절차를 거쳐 검토한다.

#### 사유

- 개인정보 외부 전송, 업무위탁, 국외이전, API 제공자의 보관·학습 정책 검토가 필요하다.
- API 키 관리 및 비용 문제가 있다.
- 현재 목표는 내부 처리 기반의 스캔본 PDF 자동 후보 생성이다.
